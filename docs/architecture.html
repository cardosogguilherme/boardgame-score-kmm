Board Game Score — Kotlin Multiplatform project guide

A Kotlin Multiplatform app for scoring board games from author-defined rule templates: build a template (fields + typed scoring rules), start a game, add players, and enter scores with a live total, per-category breakdowns, and cross-player ranked goals. The scoring engine is pure Kotlin; persistence is one shared Room layer; the UI is Compose Multiplatform. The same shared App() runs on Android today and is scaffolded to run on iPhone. This page explains how it all fits together.

1 · What the app does

Four screens, one stateless-screen / shared-ViewModel pattern throughout.

Library

Browse saved templates and games. Tap + to author a template, “New game” to start one, or a game card to resume it.

Rule builder

Name a template, add fields (COUNT/TRACK/RANKING), and add any of five scoring-rule types from a palette — each rendered as a plain-language line with live validation.

New game

Pick a saved template + optional scenario, name the game, add players. Start unlocks once a template + name are set.

Score entry

Steppers only (never text fields), a live running total, per-category subtotals, and a cross-player ranked-goal section that resolves rank live. Every edit autosaves.

2 · How the Multiplatform build is wired settings.gradle.kts · libs.versions.toml

Kotlin Multiplatform means one Gradle build with several modules, each declaring compile targets. Shared code lives in commonMain and is compiled once per target. The project is a clean one-way dependency chain: hosts → :shared → (:ui, :data) → :deepsea-scoring. Anti-corruption layers keep distinct models per layer.

Targets & source sets

  • Every module targets androidTarget() + iosX64/iosArm64/iosSimulatorArm64(); :deepsea-scoring and :ui also keep jvm() for fast unit tests.
  • Real code lives in commonMain. The only expect/actual seams are the Room DB builder and the Koin platformModule() (the Context-vs-path / clock differences).
  • Tests: domain + UI reducers run on jvm(); Room round-trips run on the Android target under Robolectric (androidUnitTest); ViewModels run as plain JVM unit tests in :shared’s commonTest.
  • iOS compiles are macOS-only — the targets are authored and the Android/JVM build is fully verified on the (Windows) authoring host.

Build wiring

  • settings.gradle.kts — includes :deepsea-scoring · :ui · :data · :shared · :androidApp; foojay auto-provisions the JDK toolchain.
  • gradle/libs.versions.toml — one version catalog (Kotlin, Compose MP, Room 2.7, Koin 4, MP navigation/lifecycle, kotlinx).
  • jvmToolchain(21) everywhere → compiled with JDK 21. The Gradle 8.14 daemon also needs JDK 21, pinned outside the repo (the host’s only system JDK is too new).
  • DI is Koin (no compiler plugin / KSP) so the graph can live in commonMain; only the platform module is per-target.
The product bet (Design law §3): scoring is an AST of dumb data + external interpreters. Rules carry no behaviour; every operation is a top-level when in the engine. Adding a rule type is a compile-error-driven change across the interpreters — that’s what lets the domain stay 100% framework-free and the UI just render it.

3 · Module responsibilities

:deepsea-scoring

Models, the scoring engine (dumb-data rules + interpreters), repository interfaces, and interactors (use cases). Zero Compose/Android. Tested on the JVM.

:data

One shared Room 2.7 (KMP) layer: entities, DAOs, the database, entity↔domain mappers, and the Room-backed repository impls. Bundled SQLite driver.

:ui

Compose-Multiplatform: immutable UiState models + domain→UI reducers + stateless screens. Never sees Room/Android types.

:shared

The app glue: StateFlow<UiState> ViewModels, the AppNavHost, the Koin graph, and the App() composable — all multiplatform.

:androidApp

Thin Android shell: MainActivity (edge-to-edge, setContent { App() }) + ScoreApp (initKoin).

iosApp

Thin SwiftUI shell wrapping MainViewController() in a UIViewControllerRepresentable. Build/run on a Mac.

9 · Data flow (one stepper tap)

user taps + on a rule row
  ▼
ScoreEntryViewModel.onIncrement(field, player) → clamps to 0..field.max, updates local StateFlow<List<PlayerInput>>, and calls UpdatePlayerValue (autosave)
  ▼  interactor → GameRepository → Room @Transaction write
uiState = scoreEntryUiState(resolved, players, active) recomputes via Scorer over a fresh ScoringContext
  ▼
per-player rules read one sheet · Ranking reads ctx.column(field) across all players → rankAward()
  ▼
StateFlow emits → ScoreEntryScreen redraws: row points, footer total, the ranked matrix — all live
The cross-player fork is the interesting bit: a ranked goal can’t be scored from one player’s sheet, so the reducer keeps the whole table and the Ranking interpreter resolves rank from everyone’s column at once. The reducer is pure, so the demo, the holder test, and production all share it.

11 · iOS status

Scaffolded, Android-verified, iOS-pending. All five increments (I1–I5) are authored: iOS targets on every module, one shared Room 2.7 KMP data layer, the ViewModels/NavHost/Koin graph + App() in :shared, a MainViewController() Compose entry point, and the iosApp Xcode project. Because Kotlin/Native + Xcode are macOS-only, the iOS compile/link is not yet verified — but the Android app and all JVM tests pass after every increment (83 / 8-8-0 holds).

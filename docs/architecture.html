Board Game Score — KMM architecture & classes

A Kotlin Multiplatform project that renders a board-game scoring form from data. The scoring engine is pure Kotlin; the UI is Compose Multiplatform; the Android app is a thin host. This page explains how the modules are wired and what every class does.

1 · How KMM is hooked up settings.gradle.kts · libs.versions.toml

Kotlin Multiplatform means one Gradle build with several modules, each declaring one or more compile targets. Code shared across targets lives in a commonMain source set and is compiled once per target. This project has three modules and a clean one-way dependency chain.

:androidApp Android application
com.android.application · MainActivity hosts the screen
│
depends on ▼
:ui Compose Multiplatform UI library
kotlin(multiplatform) + org.jetbrains.compose · targets: androidTarget() + jvm()
│
depends on ▼
:deepsea-scoring Pure-Kotlin domain
kotlin(multiplatform) + serialization · targets: androidTarget() + jvm() · no UI deps
Android-only module Multiplatform + Compose Multiplatform, framework-free

Targets & source sets

  • Both library modules declare jvm() and androidTarget().
  • The jvm() target is what runs the unit tests (jvmTest) — fast, no emulator.
  • The androidTarget() exists so :ui's Android variant can resolve :deepsea-scoring through the dependency graph. Android consumes the same commonMain code.
  • All real code lives in commonMain; there is no expect/actual yet — nothing in this project needs platform-specific implementations.
  • iOS targets are deferred (no macOS/Xcode on the build host); adding them later is a build-file-only change because the code is already common.

Build wiring

  • settings.gradle.kts — includes :deepsea-scoring, :ui, :androidApp; foojay resolver auto-provisions the JDK toolchain.
  • gradle/libs.versions.toml — single version catalog for Kotlin, serialization, Compose MP, AGP, activity-compose.
  • jvmToolchain(21) in every module → compiled with JDK 21 (the Gradle daemon also runs on JDK 21, pinned outside the repo because the host's only system JDK is 26, which Gradle 8.14 rejects).
  • gradle.properties — android.useAndroidX=true; Compose pulls AndroidX on the Android variant.
The product bet (Design law §3): scoring is an AST of dumb data + external interpreters. Rules carry no behaviour; every operation is a top-level when in the engine package. That is why the domain module can stay 100% framework-free and the UI just renders it.

2 · Module responsibilities

:deepsea-scoring

The model + the math. Serializable data classes and the scoring interpreters. Zero Compose / Android imports in source. Tested entirely on the JVM.

:ui

Screen A in Compose Multiplatform + a state holder that wraps the engine. Renders rule rows, steppers, tags, the cross-player ranked section, and the live total. Depends only on the domain.

:androidApp

The runnable Android shell. One MainActivity that calls into :ui. No business logic.

3 · Domain classes :deepsea-scoring / commonMain

Two packages: model (the AST of dumb data) and engine (the interpreters). Plus a sample with the Deep Sea template.

package model — the data

FieldId PlayerId RuleIdIds.kt Serializable @JvmInline value class wrappers over String — typed identity, zero runtime cost. Fields are always referenced by id, never by label.
FieldKind · FieldField.kt FieldKind = {COUNT, TRACK, RANKING}; Field(id, label, kind, max?) describes one input.

ScoringRule (sealed)ScoringRule.kt The AST root. Only id is on the interface. Five leaves, each @Serializable with a @SerialName and an optional per-leaf group (UI section tag):
  • Lookup(field, table) — track value indexes a points table → LOOKUP
  • PerUnit(field, points) — points per unit → ×n
  • Flat(field) — face value → FLAT
  • PerN(field, divisor) — 1 pt per N (floor) → ÷n
  • Ranking(field, awards) — cross-player rank award → RANK
PlayerInputPlayerInput.kt (id, name, values: Map<FieldId,Int>) — one player's entered numbers.
Template · ScenarioTemplate.kt A template = fields + rules + optional scenarios. A scenario overlays extra fields/rules.
ResolvedTemplate · Template.resolve()ResolvedTemplate.kt Base + chosen scenario flattened, with a labels map. This is what drives the screen.

package engine — the interpreters

Every operation over the sealed AST is an exhaustive when here — adding a rule type forces a compile error at each one. Nothing lives on the rules themselves.

SymbolFileRole
ScoringContextScoringContext.kt Holds the whole table — (template, players) — plus column(field). PER_PLAYER rules read one sheet; TABLE rules need everyone.
Scorer (object)Scorer.kt score(rule, player, ctx), breakdown(player, ctx): Map<RuleId,Int>, total(player, ctx). Total = sum of leaf rules.

validate()Validate.kt Structural checks → list of problems (empty == valid). Enforces e.g. a TABLE rule sits on a RANKING field.

4 · UI classes :ui / commonMain

A state holder wraps the engine; composables render it. The domain module is never touched — the UI only orchestrates it. Numeric input is steppers, never text fields.

SymbolFileRole

6 · Data flow (one stepper tap)

user taps + on a rule row
  ▼
ScoreEntryState.increment(field) → rebuilds the active PlayerInput (value clamped to field.max)
  ▼  Compose snapshot state changes → recomposition
sections() / total / breakdown recompute via Scorer over a fresh ScoringContext(template, players)
  ▼
per-player rules read one sheet · Ranking reads ctx.column(field) across all players → rankAward()
  ▼
ScreenA redraws: row points, footer total, and the cross-player ranked matrix — all live
The cross-player fork is the interesting bit: the ranked goal can't be scored from one player's sheet, so the state holder keeps the entire table and the Ranking interpreter resolves rank from everyone's column at once.
